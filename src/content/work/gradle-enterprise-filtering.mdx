One typed input replaced a dozen scattered dropdowns. The signal was loud: customers were exporting to CSV and filtering in Excel. A product that pushes users into spreadsheets has a filter problem, not a data problem.

(First concept: visual query builder with a clickable canvas. Too slow for experts, too opaque for beginners. Typed input served both.)

## Typed suggestions surface the taxonomy

User types "slow." The omnibar offers "duration greater than" and "status: failed." Filter types learned by using, not reading docs.

## Implicit AND across stacked filters

Most diagnostic queries are AND chains. V1 defaulted to implicit AND. Skipped the boolean builder. Most of the analytical value for a fraction of the interaction cost.

  src={gradleFilter02}

## One pattern, every diagnostic view

Same filter chain drives every downstream view: scoped task list, diagnostic detail, summary aggregations. Learned once, applied everywhere.

## What moved

Users stopped asking for CSV exports. Customer pull turned the omnibar from "the new filter on one view" into the filter pattern for the product. Every feature team adopted it.
