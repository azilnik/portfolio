Twelve dropdowns spread across a page aren't twelve choices. They're twelve interruptions. Every dropdown forces the user to stop, locate, parse options, decide, then reorient. One input with intelligent suggestions compresses that entire sequence into a single interaction point. The user types what they're thinking and the system meets them there. Fewer controls, not fewer capabilities.

## The evidence

Gradle builds developer productivity tools. Large engineering orgs like Netflix and Salesforce use Gradle Enterprise to keep CI/CD pipelines healthy. As lead designer, I owned how those teams diagnosed slow tests and tracked down flaky failures across thousands of daily builds.

The behavioral signal was loud: customers were exporting to CSV and slicing data in spreadsheets. A product that pushes users into Excel has a filter problem, not a data problem. The original concept was a visual query builder with a clickable canvas. Too slow for experts and still too opaque for beginners. A typed input made one interface serve both audiences.

## The craft

**Typed suggestions surface the taxonomy.** A new user types "slow" and the omnibar offers "duration greater than" and "status: failed" as next steps. They learn the filter types by using them, not by reading documentation. The input teaches while it works. Experts skip the suggestions and type structured queries directly. Same input, two speeds.

<FullBleedImage
  src={gradleFilter02}
  alt="Stacked filter conditions visible as pills inside the omnibar input"
/>

**Implicit AND across stacked filters.** Most diagnostic queries are AND chains: "slow tests AND failed AND in module X." V1 defaulted to implicit AND and skipped the full boolean builder. Most of the analytical value for a fraction of the interaction cost. We prototyped against Gradle's actual build event data, not fixtures. Edge cases surfaced during prototyping — huge datasets, odd chains where suggestions misfired. Engineering saw the failure modes in a prototype and moved.

**One pattern, learned once, applied everywhere.** The omnibar wasn't just a filter for one screen. The same filter chain drove every downstream view: the scoped task list, the diagnostic detail, and the summary aggregations. A pattern that feeds every view means users learn the interaction once and carry it across the product.

## The numbers

Users stopped asking for CSV exports. Customer pull is what made the cross-team rollout possible. Every feature team eventually adopted the omnibar as their filter pattern.

## The takeaway

The visual query builder we prototyped first was arguably more powerful. It let you compose arbitrary boolean logic on a canvas. Nobody used it. The typed input was less flexible in theory but faster in practice, because it matched how developers already think about filtering — as a sentence, not a flowchart. We shipped the less-capable version and it won.
